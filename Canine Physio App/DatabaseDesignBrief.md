# Canine Physiotherapy App — Database Design Brief

## 1. Project Context

This brief supports the database design component of an MSc project. The overall system consists of two applications:

1. **Canine Physio App** — A .NET MAUI mobile application (iOS and Android) used by dog owners to follow physiotherapy exercise programmes prescribed by a practitioner. The app is currently a working clickable prototype that is being evolved into the production frontend.
2. **Admin Web Application** — A backend web application (to be built) used by physiotherapy practitioners to manage patients, prescribe exercise programmes, and review progress.

The mobile app is already functional with mocked data. Two bundled JSON files currently stand in for what the admin backend will provide:

- **PhysioContent.json** — Contains the prescribed programme for a specific dog: sessions, exercises, reps, sets, videos, instructions, and a date range.
- **AppContent.json** — Contains platform-level static text (About, Terms & Conditions, warnings/disclaimers).

In the production system, `PhysioContent.json` will be replaced by an API response generated from the admin database when the owner logs in. `AppContent.json` will be managed via a simple CMS within the admin application.

## 2. Business Domain

### 2.1 Stakeholders

| Stakeholder | Role |
|-------------|------|
| **Practitioner** | A qualified canine physiotherapist who assesses dogs and prescribes exercise programmes. Practitioners use the admin web application. |
| **Owner** | The dog's owner, who uses the mobile app to carry out the prescribed exercises at home and log progress. |
| **Pet (Dog)** | The patient. Each dog receives its own tailored programme. |

### 2.2 Business Rules

1. **Practitioner–Patient relationship:** There can be multiple practitioners in the system, but each pet is assigned to exactly one practitioner at any given time. A practitioner can have many patients (one-to-many).

2. **Sessions per day:** A programme has a maximum of two physiotherapy sessions per day — **AM** (morning) and **PM** (afternoon/evening). The practitioner decides during the review whether both sessions are required or only one session per day.

3. **Programme duration:** The sessions defined in a programme repeat every day between the programme's **start date** and **end date**. The same exercises with the same reps/sets are performed each day within that window.

4. **Review cycle:** At the end of a programme period, the owner must return to the practitioner for an in-person review before any new programme can be prescribed. There is no self-service renewal; a practitioner must always create the next programme.

5. **Owner data and external CRM:** Owner personal data (name, address, contact details, billing) is held in a **separate CRM system** that the physio app and admin backend cannot access programmatically. However, the physio system must define a **unique external key** that links an owner record in the physio database to their record in the CRM. Similarly, each pet must have a **unique key** that can be cross-referenced externally.

6. **One owner, multiple pets:** An owner can have multiple dogs. Each dog can have its own independent programme with different exercises, sessions, and date ranges. For example, Owner A's Dog 1 might have an AM-only programme focusing on hindlimb strengthening, while Owner A's Dog 2 has an AM+PM programme for post-surgical mobility recovery.

7. **Access codes (activation):** A user (owner) cannot log in to the mobile app unless they have an **active access code** on the server. This code is generated by the practitioner or the admin system and provided to the owner (e.g., during their first consultation). The code activates the owner's account and associates it with their pet(s). Without a valid, active code, the login endpoint must reject the authentication attempt.

8. **Review and prescription workflow:** During a face-to-face review with the owner and pet, the practitioner:
   - Makes **assessment notes** documenting the pet's condition, progress, and any concerns.
   - Selects **exercises** from an exercise library for the pet to perform over the prescribed period.
   - Decides whether **AM and PM sessions** are needed, or only a **single daily session**.
   - Sets the **reps** and **sets** for each exercise (these may differ from the exercise library defaults and may differ between AM and PM even for the same exercise).
   - Defines the programme **start date** and **end date**.

9. **Session objective:** Each session (AM or PM) has a **session objective** — a text description of the therapeutic goal for that session (e.g., "Strengthen hindlimbs and core muscles" or "Improve mobility, balance and spinal flexibility"). This is displayed to the owner in the app as context for the exercises they are about to perform.

10. **Exercise definition:** Exercises are defined in a master library and have the following attributes:
    - **Video** — A demonstration video showing how to perform the exercise with a dog.
    - **Image** — A thumbnail/cover image for the exercise tile in the app.
    - **Instruction text** — Ordered, step-by-step written instructions.
    - **Exercise objective** — A description of the therapeutic purpose of the exercise (e.g., "Strengthens hindlimb muscles and improves sit-to-stand transition").
    - **Default reps and sets** — Baseline values that the practitioner can override when assigning the exercise to a session.

## 3. Business Process Narrative

### 3.1 Owner Onboarding

1. The owner visits the practitioner for an initial consultation with their dog.
2. The practitioner performs an assessment, makes notes, and determines a physiotherapy plan.
3. The practitioner creates the owner and pet records in the admin system (or links to existing ones via the CRM external key).
4. The practitioner generates an **access code** for the owner.
5. The owner downloads the mobile app, enters the access code, and creates an account (email + password).
6. On first login, the owner is presented with Terms & Conditions. They must accept before proceeding.
7. After acceptance, the app fetches the owner's active programme from the server and displays the exercise dashboard.

### 3.2 Programme Prescription

1. The practitioner opens the admin web application and navigates to the pet's profile.
2. They create a new programme, selecting a start date and end date.
3. They decide on single-session (AM only) or dual-session (AM + PM) days.
4. For each session, they write a **session objective** describing the therapeutic goal.
5. They browse the **exercise library** and assign exercises to each session.
6. For each exercise assignment, they set (or accept the defaults for) **reps** and **sets**. These values may differ between AM and PM even for the same exercise.
7. They add a **warning text** for the programme (e.g., "Stop immediately if your dog shows signs of pain or distress").
8. They record **assessment notes** from the review consultation.
9. They publish the programme, which makes it available for the app to pull.

### 3.3 Daily Exercise Routine (App User)

1. The owner opens the app and is taken to the **Exercises** tab showing today's session.
2. If the programme has two sessions, the app starts with the **AM session**. The owner sees exercise tiles, the session objective/description, a progress bar, and a warning disclaimer.
3. The owner taps an exercise tile to view the **exercise detail page**: summary, video thumbnail (tap to play fullscreen), step-by-step instructions, and prescribed reps/sets.
4. After performing the exercise, the owner taps **Mark Complete** to go to the **Update Progress** page.
5. On the progress page, they enter:
   - **Actual reps** completed
   - **Actual sets** completed
   - **Discomfort level** (0–10 slider)
   - **Comments** (optional free text)
6. They save, returning to the exercise dashboard where the tile is marked with a tick.
7. This repeats for each exercise in the session.
8. When all exercises are complete (or marked as skipped), the app auto-advances:
   - If AM session is done and PM exists → loads PM exercises.
   - If PM (or single session) is done → shows "All Done!" confirmation.
9. The same routine repeats every day until the programme end date.

### 3.4 Skip Functionality

The app provides two levels of skip:

- **Skip exercise:** While viewing an exercise detail, the owner can skip that specific exercise. It is marked with a red cross on the dashboard. Optional comments can be recorded.
- **Skip session:** From the main exercises page, the owner can skip the entire current session. This advances to the next session (AM → PM, or PM → next day's AM). A maximum of one day ahead is allowed — the owner cannot skip multiple days in advance.

Skips are tracked locally on the device. In the production system, they would also be submitted to the backend so the practitioner can see which exercises/sessions were skipped and why.

### 3.5 End-of-Programme Review

1. The programme end date passes. The app no longer loads new sessions.
2. The owner returns to the practitioner for an in-person review.
3. The practitioner reviews the owner's exercise logs in the admin system: completion rates, discomfort trends, skipped exercises, and comments.
4. Based on the review, the practitioner makes new assessment notes and either:
   - Prescribes a new programme (returns to §3.2), or
   - Discharges the pet from active treatment.

### 3.6 Platform Content Management

Separately from the clinical workflow, the admin system allows management of:

- **Information page content** — "About This App", "Physiotherapy" overview, "Recognising Pain" guidance.
- **Terms & Conditions** — Terms of Service, Privacy Policy, Acceptable Use Policy. Version-controlled so that updated terms trigger re-acceptance in the app.
- **Warning disclaimers** — Exercise disclaimers shown throughout the app.

## 4. First-Pass Entity Model

### 4.1 Entity Descriptions

| # | Entity | Description |
|---|--------|-------------|
| 1 | **Practitioner** | A registered physiotherapy professional who uses the admin system. Has login credentials for the admin web app. |
| 2 | **Owner** | A dog owner who uses the mobile app. Linked to the external CRM via a unique key. Has login credentials for the mobile app. |
| 3 | **Pet** | A dog (patient). Belongs to one owner. Assigned to one practitioner. Identified by a unique key cross-referenceable with the external CRM. |
| 4 | **AccessCode** | An activation code generated by a practitioner that allows an owner to register/login. Has an expiry and a status (active/used/expired). |
| 5 | **Programme** | A prescribed course of physiotherapy for a specific pet over a defined date range. Created by a practitioner after a review. |
| 6 | **AssessmentNote** | Clinical notes made by the practitioner during a review consultation. Linked to a programme (one review produces one programme). |
| 7 | **DailySession** | Defines one session slot within a programme — either AM or PM. Contains a session objective. A programme has 1 or 2 daily sessions. |
| 8 | **Exercise** | A master exercise in the library. Contains the video, image, instructions, exercise objective, and default reps/sets. Reusable across many programmes. |
| 9 | **SessionExercise** | A junction/assignment entity linking an exercise to a daily session with overridden reps and sets, and a sort order. |
| 10 | **ExerciseLog** | A progress record submitted by the owner for a specific exercise on a specific date/session. Contains actual reps, actual sets, discomfort level, comments, and timestamp. |
| 11 | **SkipLog** | A record of a skipped exercise or session, with optional comments and timestamp. |
| 12 | **AppContent** | CMS-managed text sections displayed in the app (Information pages, Terms & Conditions, warnings). |
| 13 | **TermsVersion** | Version record for Terms & Conditions, enabling the app to detect updates and require re-acceptance. |
| 14 | **TermsAcceptance** | Records that a specific owner accepted a specific terms version at a point in time. |

### 4.2 Relationships

```
Practitioner ──────────< Pet >──────────── Owner
      │                    │
      │                    │ (one practitioner per pet,
      │                    │  one owner per pet,
      │                    │  owner can have many pets)
      │                    │
      │              Programme
      │               │      │
      │        AssessmentNote │
      │                      │
      │              DailySession (1 or 2 per programme)
      │                   │
      │            SessionExercise ───────── Exercise (library)
      │                   │
      │           ┌───────┴────────┐
      │      ExerciseLog       SkipLog
      │      (per owner,       (per owner,
      │       per date,         per date,
      │       per session)      per session)
      │
AccessCode ──── Owner
```

#### Cardinality Summary

| Relationship | Cardinality | Notes |
|-------------|-------------|-------|
| Practitioner → Pet | One-to-Many | One practitioner per pet; a practitioner has many pets |
| Owner → Pet | One-to-Many | An owner can have multiple dogs |
| Pet → Programme | One-to-Many | Over time a pet will have successive programmes (one active at a time) |
| Programme → AssessmentNote | One-to-One | Each programme originates from one review; the review's notes attach to the programme |
| Programme → DailySession | One-to-Many (max 2) | One or two sessions per day (AM, PM) |
| DailySession → SessionExercise | One-to-Many | A session contains one or more exercise assignments |
| Exercise → SessionExercise | One-to-Many | A library exercise can appear in many sessions across many programmes |
| SessionExercise → ExerciseLog | One-to-Many | One log per date the exercise was performed (programme repeats daily) |
| SessionExercise → SkipLog | One-to-Many | One log per date the exercise/session was skipped |
| Practitioner → AccessCode | One-to-Many | A practitioner generates codes for their owners |
| Owner → AccessCode | One-to-One (or One-to-Many) | An owner requires at least one active code; could support re-issue |
| Owner → TermsAcceptance | One-to-Many | One acceptance per terms version |
| TermsVersion → TermsAcceptance | One-to-Many | Many owners accept each version |

## 6. Notes for Database Design

1. **ExerciseSet vs DailySession:** In the current app, `ExerciseSet` is a container nested inside `DailySession`. In the normalised database, the session objective moves to `DailySession`, and the exercises are linked via the `SessionExercise` junction table. The `ExerciseSet` entity is absorbed — it was a JSON grouping convenience, not a distinct business concept.

2. **Reps/Sets overrides:** The exercise library holds default reps/sets. When a practitioner assigns an exercise to a session, they specify the reps/sets for that particular assignment in `SessionExercise`. The same exercise can appear in both AM and PM with different reps/sets (as seen in the current mock data where "Raised sit stands" has 5 reps / 2 sets in AM but 3 reps / 2 sets in PM).

3. **Daily repetition model:** The programme does not store 14 separate days of session data. It stores 1–2 `DailySession` templates that repeat every day between `StartDate` and `EndDate`. The `ExerciseLog` and `SkipLog` tables capture per-day actuals.

4. **Video/Image hosting:** The exercise library stores filenames/URLs. In production, media assets would be hosted on a CDN. The admin system handles upload; the app downloads or streams on demand.

5. **Soft deletes:** Use `IsActive` flags rather than hard deletes for practitioners, owners, pets, and exercises. Historical data (logs, programmes) must remain intact even if the originating entity is deactivated.

6. **External CRM keys:** Both `Owner.ExternalCrmKey` and `Pet.ExternalCrmKey` are opaque strings agreed with the external CRM provider. The physio system never calls the CRM API — these keys exist solely for manual cross-referencing by admin staff.

7. **Access code flow:** The access code is a one-time (or time-limited) activation mechanism. Once redeemed, it links to the owner. The owner's `IsActive` flag and the existence of an active programme govern ongoing access — the code itself is not re-checked on every login.

8. **Programme status lifecycle:** Draft → Active → Completed. Only one programme per pet should be Active at any time. The admin system enforces this constraint.
