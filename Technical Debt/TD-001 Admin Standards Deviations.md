# TD-001 — Admin solution deviations from `Standards/coding-standards.md`

**Raised:** 2026-05-28 (end of Day 2, MSc Assessment 3 build week)
**Scope:** `Canine Physio Admin/` (ASP.NET Core admin web app deployed for the COM712 infrastructure assessment)
**Context:** The MSc submission is an **infrastructure** assessment — the marker is grading the Azure architecture, deployment, identity, secrets, observability and cost story, not the codebase. The admin app exists to give the cloud a real workload to host. After submission this codebase becomes the basis for the production Canine Physio Admin solution, at which point the deviations below must be paid down before any second feature lands.

This file replaces the per-deviation ADRs that `coding-standards.md` would otherwise require. Each row carries a remediation owner of **"post-assessment, Increment 2 kickoff"** unless stated otherwise.

---

## Decision

For the duration of MSc Assessment 3 (build complete by Sun 1 Jun 2026, submission Mon 2 Jun) the admin app is permitted to ship in its current shape. The "prototype escape hatch" in §2 of the coding standards (_"For pure prototyping (Increment 0/1 design-validation), put everything in `HelloBuddy.Web` and lift it out as boundaries crystallise"_) is invoked across the board. After submission, **no new feature work** starts until the items flagged "Increment 2 blocker" below are cleared.

The HTTP-level split into `ca-hello-buddy-ui` / `ca-hello-buddy-api` / `ca-hello-buddy-pdf` is live. The residual debt is moving PDF generation off the synchronous request path onto a Container App Job or queue-triggered worker for production scale-out — that work is now tracked as **TD-008**.                 |
| TD-005 | Replace `X-Practitioner-Id` header inter-service auth with Entra-issued workload-identity JWT validated via `AddJwtBearer`                                                                                                                                                             | DEC-010. Acceptable for Release 1 with one seeded practitioner identity and an internal-ingress network boundary; not acceptable for production once there are multiple practitioner identities or any path that could place a hostile workload inside the Container Apps Environment.                                                             |
| TD-008 | Move synchronous PDF generation off the publish request path onto a queue-backed worker (Container App Job or Azure Functions) for production scale-out                                                                                                                                | Residual scope of the originally-bundled TD-002. With DEC-009 the PDF tier is its own container but the API still blocks on `POST /render` for ~5.6 s per publish. Production should accept the publish, enqueue the render, and notify the UI when the blob lands.                                                                                |
| TD-003 | ~~Disable shared-key access on `sthellobuddyprod` (`allowSharedKeyAccess = false`) once verification tooling is Entra-only~~ **Superseded 2026-05-28 by DEC-011 + TD-007**                                                                                                             | DEC-008. The shared-key path was only needed for the Day 3 verification workflow. With DEC-011 the published-programmes container is fully Entra-only at the data plane (UAMI + user-delegation SAS), so the broader posture of disabling shared-key sits inside the production redesign captured by TD-007 rather than as a standalone debt item. |
| TD-004 | Local Docker build is blocked on this DevBox (HTTP 470 from Fastly to Debian mirrors); CI must build with `az acr build` or a non-Debian base                                                                                                                                          | DEC-006. The CI pipeline added under D-25 must either run `az acr build` (recommended) or switch to a Microsoft Container Registry base image that does not require `apt-get update` at build time.                                                                                                                                                |
| D-27   | Container App ingress is internet-facing without WAF/Front Door in front                                                                                                                                                                                                               | Acceptable for an assessment demo with a single seeded practitioner identity; for production add Azure Front Door + WAF policy and restrict the Container App ingress to the Front Door egress prefix.                                                                                                                                             |
| TD-007 | Revisit blob download path at production scale: replace per-request user-delegation SAS minting with Azure Front Door + signed URLs + content negotiation (and disable shared-key on the storage account at the same time, picking up the residual scope of the now-superseded TD-003) | DEC-011. Acceptable for assessment scale; production should not synthesise SAS per request. The current API path opens a `GetUserDelegationKey` round-trip plus a SAS sign on every download click (mitigated by an in-process key cache of ~6 days), and the resulting URL leaks the storage account hostname to the browser.                     |
