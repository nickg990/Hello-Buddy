# Hello Buddy Cloud Admin

## Azure Storage Layout Recommendation

## Purpose

This document records the recommended Azure storage layout for the Hello Buddy admin application, with a specific focus on what should be stored in MySQL, what should be stored in Azure Blob Storage, and which Blob containers should be public or private.

It is intended to support:

- assessment write-up;
- implementation planning;
- secure but simple first-increment delivery;
- consistency across future development chats.

## Recommended Approach

Use both MySQL and Azure Blob Storage, with a clear separation of concerns:

- MySQL stores structured business records and metadata;
- Blob Storage stores binary file artefacts;
- only the exercise-video container is publicly readable;
- all other containers remain private.

This gives a simple owner experience for video playback from the published PDF while keeping owner documents and operational assets locked down.

## Storage Responsibilities

### MySQL should store

- owners, pets, and treatment cases;
- draft programmes and published programme metadata;
- programme sessions and selected exercises;
- exercise titles, summaries, instructions, categories, and default prescription values;
- image metadata such as file name, blob key, alt text, and active flag;
- video metadata such as display text, URL, or blob-relative path;
- published PDF metadata such as file name, blob key, version number, generated date, and generated by.

### Blob Storage should store

- published programme PDF files;
- exercise thumbnail or reference images;
- exercise demonstration video files if the solution hosts them directly;
- optional future document artefacts or exports.

## Recommended Storage Account Model

### Storage account

Recommended name:

- `sthellobuddyprod`

Configuration intent:

- General Purpose v2;
- hot tier for first increment;
- public blob access allowed at account level only because one specific container needs anonymous read access;
- all containers private by default unless explicitly opened.

Important note:

Allowing public blob access at the storage-account level does not make all blobs public. It only permits container-level public access where explicitly configured. The design should still keep all non-video containers private.

## Recommended Container Layout

| Container name         | Access level | Purpose                                                | Notes                                                                    |
| ---------------------- | ------------ | ------------------------------------------------------ | ------------------------------------------------------------------------ |
| `published-programmes` | Private      | Stores final published programme PDFs                  | Customer-facing documents should not be anonymously accessible.          |
| `exercise-images`      | Private      | Stores exercise thumbnails or supporting images        | Accessed via the application, not directly by anonymous users.           |
| `exercise-videos`      | Public read  | Stores demonstration videos linked from published PDFs | Chosen for first-increment simplicity so owners can open links directly. |
| `internal-assets`      | Private      | Stores any admin-only or working artefacts             | Optional, but useful to keep internal files separate.                    |

## Why only videos should be public

The published PDF is intended to be sent to pet owners, and the video links inside it must work without requiring the owner to log into the admin system.

For the first increment, the simplest solution is:

- host exercise videos in a dedicated Blob container;
- configure that one container for anonymous read access;
- store public links to those videos in the exercise records;
- keep PDFs and all other assets private.

This means:

- owners can click a video link in the PDF and play the video immediately;
- no owner data is publicly exposed;
- no published programme PDFs are publicly listed or directly exposed by default;
- the storage design remains easy to explain in the assessment.

## Why text should not be in Blob Storage

Exercise text should stay in MySQL rather than Blob Storage.

That includes:

- exercise title;
- summary;
- instructions;
- practitioner notes;
- contraindications;
- category and status;
- default reps, sets, hold, and frequency values.

Reason:

- the application needs this text to be searchable, editable, validated, and reusable inside the programme builder and PDF generation flow.

Blob Storage is appropriate for files, not for the main structured content of the exercise library.

## First Increment Recommendation

For the assessment increment, use the following practical arrangement:

- MySQL for all structured application data;
- Blob container `published-programmes` for final PDFs;
- Blob container `exercise-images` for thumbnail images;
- Blob container `exercise-videos` for publicly accessible demonstration videos;
- MySQL records storing blob keys and public video URLs where needed.

This aligns with the current first-increment flow:

- open case;
- build draft programme;
- preview inside the page;
- publish immutable version;
- store final PDF in private Blob storage;
- allow owners to open public video links from the issued PDF.

## Security Position

This design is intentionally selective rather than fully public.

Public:

- exercise demonstration videos only.

Private:

- published PDFs;
- draft assets;
- internal artefacts;
- structured records in MySQL;
- all application services except the UI.

This gives a defensible security story for the MSc assessment:

- the public surface is limited to non-sensitive media designed for external viewing;
- owner-specific documents remain controlled;
- operational and clinical records remain behind the application boundary.

## Assessment Rationale

This recommendation is a good fit for the assessment because it balances:

- technical clarity;
- sensible security boundaries;
- realistic deployment decisions;
- low implementation friction for the first increment.

It is stronger than putting all assets into one public container and simpler than building a signed-URL or streaming-authorisation model for the first release.

## Later Hardening Options

If the project evolves beyond the first increment, possible next steps include:

- moving exercise videos behind short-lived signed URLs;
- serving media through the API or an edge layer;
- separating production and non-production storage accounts;
- applying lifecycle policies and content delivery optimisation.

Those changes are not required for the first assessment increment.
